Grand Line AI Trainer (GLAT)
Project Design Doc

Last updated: 2026-04-26

1. Project Goal

Build a One Piece Card Game training assistant where:
- the rules engine is the source of truth
- the player reports their real-life actions into the app
- the AI tracks state and chooses its own legal actions
- no illegal moves are ever executed by the AI

Primary intended product experience:
- the human plays their real physical deck at the table
- the human reports what they played and what happened during their turn
- the program updates the shared board state from those reports
- the AI then plays its own turn through the engine using only legal actions
- over time, the reporting UX should become simple enough that the player can report compact actions like:
  - "I played OP12-012"
  - "I attacked your leader with OP12-094"
  - "I used a 2k counter on Koala"

The long-term architecture should be:
- Engine enforces rules and state transitions
- Human input layer records what the player did in real life
- LLM chooses among engine-generated legal actions only
- Future review layer analyzes games and produces lessons
- Future training modes can also run AI vs AI games entirely inside the engine

Primary product modes:
- Physical Table Mode
  - the human uses their real deck, playmat, hand, and life cards outside the program
  - the physical table is authoritative for the human side
  - the app accepts reported facts like "play OP12-021" even when that card was not preloaded into a tracked human decklist
  - the app looks up known card IDs, creates tracked card instances as needed, and asks for clarification or manual resolution when effects are unsupported
  - the AI side remains strict: AI actions must come from engine-generated legal actions and must validate before execution
- Digital Test Mode
  - both players' decks and hidden zones are represented in engine state
  - commands are stricter because the engine can verify hand contents, deck contents, and legality
  - this remains the main path for regression tests, GUI testing, replay/debugging, and future AI vs AI self-play

Important mode boundary:
- physical reporting should be flexible for the human side because the real game already happened on the table
- AI play must stay engine-strict in every mode
- unsupported human card effects should never be invented by the LLM; they should be skipped, manually resolved, or tracked as notes until implemented


2. Current Status

What is already built:
- cards.txt -> cards.json deck extraction pipeline
- local deck/card data loading from the One Piece card JSON files
- a Phase 1 deterministic OPCG engine in glat_engine.py
- turn phases: refresh, draw, don, main, end
- validation-first action system
- combat resolution for the simplified Phase 1 ruleset
- action logging
- automated tests for turn progression, illegal action rejection, legal-only actions, and combat
- Gemini integration for Phase 2
- legal action generation in referee.py
- AI main phase flow where:
  - engine generates legal actions
  - Gemini selects from that list using one call per turn
  - Gemini can return a short ordered turn plan
  - engine validates and executes each chosen step
  - engine safely falls back to end_turn when needed
- both P1 and P2 decks are built from cards.json
- guided CLI game loop in cli_game.py for real local play/testing
- local operator GUI in operator_gui.py for match control and intake/testing
- first-pass table-shaped GUI with visible zones, compact card tiles, hidden life display, click-to-inspect card details, replay navigation, and horizontally scrollable card zones
- save/load and mid-game resume support for CLI engine games
- manual state/effect helpers for:
  - draw
  - trash from deck
  - reveal top cards
  - discard
  - KO
  - move between zones
  - add life
- initial deck setup now tracks real life cards, not only a life integer
- initial setup now shuffles decks, draws the opening hand before placing life, and supports one opening mulligan before life is placed
- first-pass deterministic effect framework for the current Koala deck
- human interactive effect choices in the CLI for search/select effects
- first trigger/counter helpers for current deck cards
- scenario-based human turn reporting flow for real physical play
- shorthand and chained reporting for common opponent actions
- intake session memory for follow-up references like same attacker / same target / other one
- guided review/correction tools for reconciling state drift
- explicit battle-context stages for attacks, including blocker / counter / trigger / cleanup timing
- battle-trace surfacing in the CLI, intake log, and local operator GUI
- replay/debug timeline with before/after snapshots and state diffs per applied action
- AI debug history for legal actions, planned actions, executed actions, and fallback end_turn behavior
- CLI replay/debug commands for recent replay entries, latest diff, and recent AI debug
- operator GUI replay timeline, latest diff, and AI debug panels
- pressure-test scenarios for full reported turns, blocker redirects, and trigger branches
- current-scope engine-completeness sweep for supported reaction/effect cards
- expanded regression coverage in tests/test_glat_engine.py
- dedicated CLI/operator GUI helper tests for intake and operator flows

Important implementation note:
- Phase 2 still respects the one-LLM-call-per-turn rule
- the model does not invent actions or resolve rules
- the engine remains the single source of truth
- the current AI path supports short planned sequences
- defensive play now has an explicit engine-managed battle flow for the supported scope
- the remaining gaps are mostly replay/debugging depth, operator UX, and broader card/timing coverage
- the next product direction is to separate strict digital testing from flexible physical-table reporting


3. Current Architecture

cards.json
- stores the chosen deck list and card details
- remains the core deck-definition source for strict digital games and AI deck construction
- should not be the only source for physical-table human card reports; physical mode needs lookup access to the broader local card database

glat_engine.py
- main game engine
- owns state schema, turn loop, action validation, action execution, combat, logs
- now also owns:
  - life-card tracking
  - effect resolution hooks
  - deck-specific on-play and on-KO effects
  - trigger/counter helper actions
  - explicit battle-context lifecycle during attack resolution
  - explicit cleanup/finalization for battle resolution
  - battle-trace data for CLI/GUI/debug surfacing
  - replay-log snapshots and compact state diffs per applied action
  - AI planning/debug history for local inspection
  - timing validation for live counter/trigger windows
  - optional effect-choice provider hook for human-controlled choices
  - shuffled opening setup with hand-before-life ordering and optional mulligan flags

Future match-mode state
- should explicitly record whether the current match is:
  - digital_strict
  - physical_reported
- this will control whether human reports must come from tracked hand/deck zones or can create/confirm cards from physical-table reports

referee.py
- currently contains legacy state helper code from the earlier prototype
- now also contains get_legal_actions(state, engine), which is important for Phase 2
- likely needs cleanup later so engine helpers are not split awkwardly across files

ai/agent.py
- Gemini wrapper
- formats legal actions for the model
- prompts the model to choose only from listed actions
- returns either an action index or a short turn plan

tests/test_glat_engine.py
- current verification harness
- should continue expanding as engine complexity grows

cli_game.py
- current primary operator interface
- supports:
  - AI turns through engine + Gemini/fake AI
  - guided human turn entry
  - guided opponent-turn intake flow
  - scenario-based physical turn reporting
  - raw command fallback
  - shorthand/chained report entry
  - manual effect resolution
  - interactive human effect choices
  - trigger/counter activation helpers
  - review/correction tooling
  - replay/diff/AI-debug inspection commands
  - save/load resume flow

cli_intake.py
- extracted intake/session-memory/shorthand parsing layer
- owns:
  - opponent intake session lifecycle
  - shorthand parsing and chaining
  - conversational follow-up references
  - intake-oriented command resolution helpers
  - battle-trace formatting and intake-event logging

operator_gui.py
- local table/operator panel for:
  - state inspection
  - AI turn execution
  - shorthand intake entry
  - table-shaped zone display for leaders, characters, hand, life, deck/trash, and DON
  - hidden life display so unrevealed life card identities are not leaked
  - click-to-inspect card details
  - horizontally scrollable card zones for full hands and boards
  - replay timeline review
  - latest state diff review
  - AI planning/debug review
  - latest battle trace review
  - save/load and match control

Future match command console
- should become the primary physical-table intake surface
- can be either embedded in the GUI or opened as a second window
- should feel like a match chat / command log:
  - user enters compact commands
  - system confirms accepted structured actions
  - system asks follow-up questions when input is ambiguous
  - system records manual effect notes and corrections
  - AI actions and battle results appear in the same timeline
- this should remain a structured command console, not an unrestricted rules-solving chatbot

Older prototype files
- board_gui.py
- table_gui.py
- game_state.json

These were useful for earlier experimentation but are no longer the main path for the rules engine.


4. Recommended Roadmap

Phase A: Stabilize Engine Core
Status: In progress / mostly done for simplified rules

Goals:
- keep engine as the single source of truth
- preserve legality at all times
- continue expanding tests before adding more intelligence

Needs:
- move legal action generation closer to engine ownership, or clearly define why it stays in referee.py
- clean up duplicated prototype logic so the project has one clear game-state path
- decide on one canonical saved-state file for engine games
- continue improving replay/debug serialization and inspection depth


Phase B: Full AI Turn Selection
Status: Implemented for current scope

Current behavior:
- one Gemini call per turn
- engine generates legal actions once for the turn
- Gemini returns a short ordered turn plan
- engine revalidates each action before execution
- engine ends turn safely if the plan becomes stale or incomplete

Remaining gaps:
- AI does not yet make defense or counter decisions
- AI does not yet get prompted with structured effect-choice branches during complex card resolution
- legal action generation still lives outside the engine file


Phase C: Human Move Intake
Status: Strong implementation for current operator/testing scope

What exists now:
- guided CLI turn flow for the human player
- guided opponent-turn intake flow in the CLI for real physical play
- local operator GUI for report-driven testing and match control
- raw command fallback
- manual state correction and effect-resolution tools
- interactive prompts for human-controlled effect choices on supported cards
- scenario-driven reporting prompts for common real-table actions
- life-damage reporting helpers for physical play
- shorthand reporting for play / attack / attach / counter / trigger / damage / discard / state changes
- chained shorthand commands
- session memory for follow-up phrases

Goal:
- let the human play physically and report their moves
- engine updates internal game state from those reports
- make the reporting loop clean enough that the AI can react to a reported physical turn without manual reconstruction work
- support a physical-table mode where the human can report real cards from outside a preloaded engine decklist, while still preserving strict AI legality

Needed features:
- explicit match mode flag for digital_strict vs physical_reported
- physical-mode card lookup and dynamic human card instance creation from any known card ID
- command-console UX for report entry, follow-up prompts, confirmations, and manual effect notes
- richer prompt-driven flow for reporting real-life opponent turns
- support for effect resolution that depends on hidden zones or player intent
- correction/reconciliation flow when the physical game and engine state diverge
- eventually, optional natural-language parsing on top of the structured flow
- ongoing shorthand/reporting refinement so the player can report actions primarily by card ID or short text commands

What is still missing:
- better hidden-information handling when the human intentionally wants to keep some details unknown to the AI
- flexible physical-table reporting that can accept cards not present in the tracked human hand/deck
- explicit unsupported-card/effect handling so the app can track a card body without pretending to resolve unimplemented rules text
- richer attack follow-up prompts and clearer correction/reconciliation UX
- stronger GUI-native intake flows so the operator does not need to rely on text entry for everything
- more pressure-testing across long real games to expose reporting friction and drift patterns

Recommended approach:
- do not use free-form natural language first
- start with a constrained command syntax in a match command console, plus button helpers for common actions
- later add an LLM parser for convenience after the structured path is stable
- preserve a strict structured path underneath any future shorthand or natural-language convenience layer
- in physical_reported mode, let the human report a played card by ID even if the engine did not already track it in hand
- in digital_strict mode, require cards to exist in the proper tracked zone before they can be played
- every unsupported effect should resolve through one of:
  - skip effect
  - manual zone/state changes
  - record note only
  - implement the resolver later


Phase D: Defense / Reaction System
Status: In progress, but much closer to stable for the current supported scope

What exists now:
- manual counter helper for current deck cards
- manual trigger helper for current deck cards
- counter events are restricted to opponent-turn defensive timing and cannot be played as normal cards during your own turn
- attack resolution now maintains an explicit battle context in engine state and logs the current step
- attack declaration now includes named stages for:
  - blocker window
  - counter window
  - damage resolution
  - trigger handling on life damage
  - KO resolution
  - cleanup
- manual counter/trigger helpers now validate against live battle windows when a battle context exists
- battle traces are surfaced into the CLI, intake history, and operator GUI
- human defense prompts in CLI
- deterministic engine-controlled AI defense without extra LLM calls

Still needed for real gameplay:
- broader supported edge cases across battle branches for the active deck pool
- richer defender-choice handling where supported effects create branching decisions
- fuller timing coverage if the supported deck pool expands
- optional AI defensive choices
- proper battle timing across a wider set of future reactions and cards

Clarification:
- a real engine-managed reaction flow now exists
- what remains is depth/completeness, not starting from zero

Important note:
- this was the biggest gameplay gap and is now in stabilization territory for the current scope
- future work here should be targeted completeness work, not another major structural rewrite


Phase E: Card Effects
Status: In progress with a stronger current-scope effect baseline

Implemented so far:
- effect framework inside glat_engine.py
- supported on-play / on-KO / leader effects for current deck cards:
  - OP12-097 Captains Assembled
  - OP12-086 Koala
  - OP12-094 Monkey.D.Dragon
  - OP12-087 Nico Robin
  - OP12-119 Bartholomew Kuma
  - OP14-108 Silvers Rayleigh
  - OP12-089 Hack
  - OP10-109 Basil Hawkins
  - EB03-042 Koala
  - EB03-053 Emporio.Ivankov
  - OP12-081 Koala Leader
- supported trigger/counter helpers for:
  - OP12-098 Hair Removal Fist
  - OP06-115 You're the One Who Should Disappear.
  - OP12-112 Baby 5
- recent completeness/testing passes now explicitly cover:
  - EB03-053 on-play behavior
  - OP10-109 on-KO life trash
  - EB03-042 on-KO character play
  - OP06-115 trigger timing
  - manual trigger timing inside a live trigger window

Still missing:
- many additional card texts if the supported deck pool changes
- more generic timing-window coverage outside the current tested card set
- deeper optional-effect handling as more branching effects are added

Needed:
- effect framework
- triggered effects
- on play / on attack / on KO handlers
- card-specific resolvers
- future retrieval or lookup layer for effect text

Recommended direction:
- do not let the LLM directly resolve rules text
- use the LLM only for strategic choice
- actual card effects should be implemented as engine logic over time


Phase F: Product UI
Status: Local table/operator GUI implemented and moving toward a physical-table command-console workflow

Original idea:
- Streamlit or lightweight Python UI

What exists now:
- terminal-driven guided app in cli_game.py
- local table/operator GUI in operator_gui.py with:
  - current state snapshot
  - AI turn execution
  - shorthand human/opponent report input
  - table-shaped board zones
  - compact clickable card tiles
  - hidden life display
  - scrollable hand/board/life zones
  - card details panel
  - replay back/forward/return-live controls
  - replay timeline panel
  - latest state diff panel
  - AI debug panel
  - battle trace panel
  - intake/action log views
  - save/load and match control

Current recommendation:
- keep the local GUI focused on real match operation, physical-table reporting, and testing
- build a match command console before chasing polished visuals or card art
- keep the central board readable enough to validate state, but let the command console carry most intake and clarification work

Next UI option later:
- richer local board UI and card art once physical reporting and correction flows are reliable

Command console still needed:
- display the match as a chat-like action timeline
- accept compact structured commands like:
  - play OP12-021
  - attach 1 leader
  - attack OP12-094 leader
  - counter OP12-098 leader
  - take 1 life
- ask follow-up questions for ambiguous cards, targets, hidden-zone effects, and unsupported effects
- support physical_reported mode where human-side cards can be instantiated from a card ID lookup
- support digital_strict mode where commands must reference tracked hand/board/life/deck state
- let AI actions, battle windows, trigger choices, corrections, and replay markers appear in the same timeline

Replay viewer still needed:
- step through saved games one action at a time instead of only viewing the latest replay entries
- select a replay step and inspect the full before/after snapshot, not just the compact diff summary
- filter by turn, player, or action type so long games stay navigable
- jump directly from a replay step into the related battle trace or AI debug entry
- support loading an older saved state and browsing replay history without advancing the game

The old playmat UI can be reused later if desired, but it should not be the main focus now.


Phase G: Learning / Review Loop
Status: Future

Goal:
- analyze completed games
- identify mistakes
- produce lessons and strategy notes

Useful outputs:
- "AI overcommitted DON on turn 3"
- "AI missed lethal"
- "AI attacked into inefficient counters"

Recommended future implementation:
- save full action logs and final state
- run a post-game reviewer over the log
- store lessons in a text file or JSON memory store
- feed only a very small number of recent lessons into future prompts


Phase H: Training Modes
Status: Future

Target modes:
- Human vs AI with the human using a real physical deck and reporting actions into the app
- AI vs AI self-play for testing, regression hunting, and strategy training

Notes:
- Human vs AI remains the primary user-facing mode
- AI vs AI should reuse the same engine/state/logging path whenever possible
- self-play should be introduced only after the human-reporting path is stable enough to trust as the main product loop


5. Priority Order From Here

Recommended next implementation order:

Current updated order after the intake/parser/operator-GUI and battle-stabilization milestones:
- 1. Add explicit match mode support: digital_strict vs physical_reported
- 2. Build the match command console for physical-table reporting, confirmations, and follow-up prompts
- 3. Add physical-mode card lookup and dynamic human card instance creation from any known card ID
- 4. Add unsupported-effect handling paths: skip, manual resolve, note only, implement later
- 5. Improve correction/reconciliation UX for physical table drift
- 6. Continue replay/debug work by making replay entries selectable and tied to board snapshots
- 7. Move legal action generation closer to engine ownership or formalize its boundary
- 8. Continue targeted card-effect support only where needed for the chosen deck or new test scenarios
- 9. Improve AI planning around effect choices, offensive sequencing, and future defensive choices
- 10. Add post-game review and lesson tracking
- 11. Add AI vs AI self-play/training modes on top of the same engine



6. Technical Risks

Main risks:
- splitting core logic across glat_engine.py and referee.py can become confusing
- letting the LLM do too much rules reasoning will create illegal-state risk
- human free-text input will be messy unless constrained first
- adding card effects without a clear effect framework will create technical debt
- reactive battle timing can become brittle if layered onto the current simplified attack flow without a clearer step model
- physical_reported mode can create state drift if reported facts are accepted without good confirmation, correction, and audit trails
- allowing any card ID from the full database can mislead users unless unsupported effects are clearly marked and manually resolved

Guiding principle:
- correctness > intelligence
- engine authority > model creativity
- structured input > free-form parsing
- physical table authority for human reports, engine authority for AI actions


7. Recommended File Direction

Suggested future structure:

glat_engine.py
- keep or split into engine/, but this should remain the main source of truth

ai/agent.py
- keep model integration here

referee.py
- either:
  - narrow it to utility functions like legal action generation, or
  - retire it and move legal action generation into engine/

tests/
- continue expanding around every new rules feature

docs/
- later move this file there if the project grows


8. Definition of Near-Term Success

The next meaningful milestone should be:

"The player can use a real physical deck, report an entire turn through a command-console flow, resolve unsupported effects manually when needed, correct drift safely, and let the AI respond from the resulting state without reconstruction work."

That is the point where GLAT becomes a usable real-world training tool instead of only a simulation prototype.

Current assessment:
- GLAT is now close to that threshold for the currently supported deck/testing scope
- the engine is no longer the obvious primary bottleneck
- the biggest remaining work is around replay/debug confidence, operator UX, and broadening supported gameplay coverage carefully

Longer-term success looks like:

"A human can play a real deck against the AI with low-friction reporting, and the same engine can also run AI vs AI games for testing and training."
