Grand Line AI Trainer (GLAT)
Project Design Doc

Last updated: 2026-04-29

1. Project Goal

Build a One Piece Card Game training assistant where:
- the rules engine is the source of truth
- the player reports their real-life actions into the app
- the AI tracks state and chooses its own legal actions
- no illegal moves are ever executed by the AI

Primary intended product experience:
- the human plays their real physical deck at the table
- the human reports what they played and what happened during their turn
- the program updates the shared board state from those reports
- the AI then plays its own turn through the engine using only legal actions
- over time, the reporting UX should become simple enough that the player can report compact actions like:
  - "I played OP12-012"
  - "I attacked your leader with OP12-094"
  - "I used a 2k counter on Koala"

The long-term architecture should be:
- Engine enforces rules and state transitions
- Human input layer records what the player did in real life
- LLM chooses among engine-generated legal actions only
- Future review layer analyzes games and produces lessons
- Future training modes can also run AI vs AI games entirely inside the engine

Primary product modes:
- Physical Table Mode
  - the human uses their real deck, playmat, hand, and life cards outside the program
  - the physical table is authoritative for the human side
  - the app accepts reported facts like "play OP12-021" even when that card was not preloaded into a tracked human decklist
  - the app looks up known card IDs, creates tracked card instances as needed, and asks for clarification or manual resolution when effects are unsupported
  - the AI side remains strict: AI actions must come from engine-generated legal actions and must validate before execution
- Digital Test Mode
  - both players' decks and hidden zones are represented in engine state
  - commands are stricter because the engine can verify hand contents, deck contents, and legality
  - this remains the main path for regression tests, GUI testing, replay/debugging, and future AI vs AI self-play

Important mode boundary:
- physical reporting should be flexible for the human side because the real game already happened on the table
- AI play must stay engine-strict in every mode
- unsupported human card effects should never be invented by the LLM; they should be skipped, manually resolved, or tracked as notes until implemented

Primary product surface direction:
- move the main user-facing experience toward a local web-based match cockpit
- the cockpit should be command-first, not drag-and-drop-first
- the board view should be a readable live state mirror, while the chat/command log should be the primary way to play
- ideal layout:
  - half or more of the page shows the board state, zones, leaders, characters, DON, life, deck/trash, and selected card details
  - the remaining side shows a chat-style match log and command input
  - pending choices, counters, triggers, blockers, and effect prompts appear as clickable buttons/chips in the chat panel
  - replay/debug panels can live behind tabs or collapsible sections
- the current Tkinter operator GUI should be treated as a prototype/debug surface, not the final UX target
- the engine, command parser, intake logic, replay logs, correction commands, and AI strictness should be preserved and wrapped by the web cockpit rather than rewritten from scratch

3. Current Architecture

cards.json
- stores the chosen AI deck list and card details
- remains the default deck-definition source for AI deck construction and P1 strict digital setup
- should not be the only source for physical-table human card reports; physical mode needs lookup access to the broader local card database

player_cards.json
- stores the chosen player/default strict-mode deck list and card details
- lets the human/default seat keep a stable deck while the AI deck is changed for heuristic testing
- is generated from player_cards.txt by the same deck extraction script
- if player_cards.json is missing, the engine falls back to cards.json for backward compatibility
- CLI, operator GUI, and web cockpit entry points accept --cards and --player-cards overrides for explicit deck pairing tests

glat_engine.py
- main game engine
- owns state schema, turn loop, action validation, action execution, combat, logs
- now also owns:
  - life-card tracking
  - effect resolution hooks
  - deck-specific on-play and on-KO effects
  - trigger/counter helper actions
  - explicit battle-context lifecycle during attack resolution
  - explicit cleanup/finalization for battle resolution
  - battle-trace data for CLI/GUI/debug surfacing
  - replay-log snapshots and compact state diffs per applied action
  - AI planning/debug history for local inspection
  - timing validation for live counter/trigger windows
  - optional effect-choice provider hook for human-controlled choices
  - shuffled opening setup with hand-before-life ordering and optional mulligan flags

Match-mode state
- explicitly records whether the current match is:
  - digital_strict
  - physical_reported
- this will control whether human reports must come from tracked hand/deck zones or can create/confirm cards from physical-table reports

referee.py
- currently contains legacy state helper code from the earlier prototype
- now also contains get_legal_actions(state, engine), which is important for Phase 2
- likely needs cleanup later so engine helpers are not split awkwardly across files

ai/agent.py
- Gemini wrapper
- formats legal actions for the model
- prompts the model to choose only from listed actions
- returns either an action index or a short turn plan

tests/test_glat_engine.py
- current verification harness
- should continue expanding as engine complexity grows

cli_game.py
- current primary operator interface
- supports:
  - AI turns through engine + Gemini/fake AI
  - guided human turn entry
  - guided opponent-turn intake flow
  - scenario-based physical turn reporting
  - raw command fallback
  - shorthand/chained report entry
  - manual effect resolution
  - interactive human effect choices
  - trigger/counter activation helpers
  - review/correction tooling
  - replay/diff/AI-debug inspection commands
  - save/load resume flow

cli_intake.py
- extracted intake/session-memory/shorthand parsing layer
- owns:
  - opponent intake session lifecycle
  - shorthand parsing and chaining
  - conversational follow-up references
  - intake-oriented command resolution helpers
  - battle-trace formatting and intake-event logging

operator_gui.py
- local table/operator panel for:
  - state inspection
  - AI turn execution
  - shorthand intake entry
  - table-shaped zone display for leaders, characters, hand, life, deck/trash, and DON
  - hidden life display so unrevealed life card identities are not leaked
  - click-to-inspect card details
  - horizontally scrollable card zones for full hands and boards
  - replay timeline review
  - latest state diff review
  - AI planning/debug review
  - latest battle trace review
  - save/load and match control
- important direction note:
  - keep operator_gui.py useful for debugging and local engine inspection
  - do not keep investing in it as the primary product interface if a web cockpit can serve the physical-table workflow more cleanly

Match command console
- is now a separate GUI window and should continue becoming the primary physical-table intake surface
- feels like a match chat / command log:
  - user enters compact commands
  - system confirms accepted structured actions
  - system asks follow-up questions when input is ambiguous or an in-app choice is needed
  - system records manual effect notes and corrections
  - AI actions and battle results appear in the same timeline
- this should remain a structured command console, not an unrestricted rules-solving chatbot

Future web match cockpit
- recommended next product surface
- should be a thin local web app over the existing Python engine and command/intake layers
- suggested backend:
  - FastAPI or Flask
  - one in-memory match/session for first pass
  - endpoints like GET /state, POST /command, POST /choice, GET /replay
  - all commands route through existing structured command/intake logic where possible
  - all AI actions continue to route through engine-generated legal actions
- suggested frontend:
  - board view on the left/center
  - chat-style command log and input on the right
  - pending prompts shown as buttons/chips in the chat panel
  - card details by click/hover
  - replay/debug available through tabs or collapsible panels
- first pass should prioritize real physical play usability over visual polish or card art
- this should become the main path for the final user experience if it proves smoother than Tkinter

Older prototype files
- board_gui.py
- table_gui.py
- game_state.json

These were useful for earlier experimentation but are no longer the main path for the rules engine.


4. Recommended Roadmap

Phase A: Stabilize Engine Core
Status: In progress / mostly done for simplified rules

Goals:
- keep engine as the single source of truth
- preserve legality at all times
- continue expanding tests before adding more intelligence

Needs:
- move legal action generation closer to engine ownership, or clearly define why it stays in referee.py
- clean up duplicated prototype logic so the project has one clear game-state path
- decide on one canonical saved-state file for engine games
- continue improving replay/debug serialization and inspection depth


Phase B: Full AI Turn Selection
Status: Implemented for current scope

Current behavior:
- one Gemini call per turn
- engine generates legal actions once for the turn
- Gemini returns a short ordered turn plan
- engine revalidates each action before execution
- engine ends turn safely if the plan becomes stale or incomplete

Remaining gaps:
- AI does not yet make defense or counter decisions
- AI does not yet get prompted with structured effect-choice branches during complex card resolution
- legal action generation still lives outside the engine file


Phase C: Human Move Intake
Status: Strong implementation for current operator/testing scope

What exists now:
- guided CLI turn flow for the human player
- guided opponent-turn intake flow in the CLI for real physical play
- local operator GUI for report-driven testing and match control
- raw command fallback
- manual state correction and effect-resolution tools
- interactive prompts for human-controlled effect choices on supported cards
- scenario-driven reporting prompts for common real-table actions
- life-damage reporting helpers for physical play
- shorthand reporting for play / attack / attach / counter / trigger / damage / discard / state changes
- chained shorthand commands
- session memory for follow-up phrases

Goal:
- let the human play physically and report their moves
- engine updates internal game state from those reports
- make the reporting loop clean enough that the AI can react to a reported physical turn without manual reconstruction work
- support a physical-table mode where the human can report real cards from outside a preloaded engine decklist, while still preserving strict AI legality

Needed features:
- continue refining explicit match mode behavior for digital_strict vs physical_reported
- continue refining physical-mode card lookup and dynamic human card instance creation from any known card ID
- continue refining command-console UX for report entry, follow-up prompts, confirmations, and manual effect notes
- richer prompt-driven flow for reporting real-life opponent turns
- support for effect resolution that depends on hidden zones or player intent
- correction/reconciliation flow when the physical game and engine state diverge
- eventually, optional natural-language parsing on top of the structured flow
- ongoing shorthand/reporting refinement so the player can report actions primarily by card ID or short text commands

What is still missing:
- better hidden-information handling when the human intentionally wants to keep some details unknown to the AI
- deeper flexible physical-table reporting beyond the first supported play-by-ID path
- deeper unsupported-card/effect handling so the app can track a card body without pretending to resolve unimplemented rules text
- richer attack follow-up prompts and clearer correction/reconciliation UX
- stronger GUI-native intake flows so the operator does not need to rely on text entry for everything
- more pressure-testing across long real games to expose reporting friction and drift patterns

Recommended approach:
- do not use free-form natural language first
- start with a constrained command syntax in a match command console, plus button helpers for common actions
- later add an LLM parser for convenience after the structured path is stable
- preserve a strict structured path underneath any future shorthand or natural-language convenience layer
- in physical_reported mode, let the human report a played card by ID even if the engine did not already track it in hand
- in digital_strict mode, require cards to exist in the proper tracked zone before they can be played
- every unsupported effect should resolve through one of:
  - skip effect
  - manual zone/state changes followed by manual done
  - record note only
  - implement the resolver later


Phase D: Defense / Reaction System
Status: In progress, but much closer to stable for the current supported scope

What exists now:
- manual counter helper for current deck cards
- manual trigger helper for current deck cards
- counter events are restricted to opponent-turn defensive timing and cannot be played as normal cards during your own turn
- attack resolution now maintains an explicit battle context in engine state and logs the current step
- attack declaration now includes named stages for:
  - blocker window
  - counter window
  - damage resolution
  - trigger handling on life damage
  - KO resolution
  - cleanup
- manual counter/trigger helpers now validate against live battle windows when a battle context exists
- battle traces are surfaced into the CLI, intake history, and operator GUI
- human defense prompts in CLI
- deterministic engine-controlled AI defense without extra LLM calls

Still needed for real gameplay:
- broader supported edge cases across battle branches for the active deck pool
- richer defender-choice handling where supported effects create branching decisions
- fuller timing coverage if the supported deck pool expands
- optional AI defensive choices
- proper battle timing across a wider set of future reactions and cards

Clarification:
- a real engine-managed reaction flow now exists
- what remains is depth/completeness, not starting from zero

Important note:
- this was the biggest gameplay gap and is now in stabilization territory for the current scope
- future work here should be targeted completeness work, not another major structural rewrite

The old playmat UI can be reused later if desired, but it should not be the main focus now.

Status: Future

Phase I: Heuristic Planning Agent
Status: Active / Core heuristic, effect-choice, opponent-awareness, and shallow rollout passes implemented

Core direction:
- use a deterministic HeuristicPlanningAgent as a separate game-AI policy
- this agent does not use an LLM API as its core move selector
- the heuristic agent receives engine state plus engine-generated legal actions, scores those legal actions, and chooses only from that list
- the rules engine/referee remain authoritative for legal action generation and state mutation
- an LLM can remain optional for explanation, coaching, post-game review, or developer assistance, but should not be required for normal AI turn execution

Reasoning:
- LLMs are useful for language, summaries, and strategic explanations, but they are slow, inconsistent, and expensive for exact game-tree decisions
- a purpose-built heuristic/search policy is easier to debug, cheaper to run locally, and better aligned with strict legal-action execution
- a custom trained transformer is premature until the project has a large dataset of states, legal actions, chosen actions, and outcomes from real or self-play games
- the right near-term path is deterministic heuristics first, then search/self-play data, then possibly a learned policy or value model later

Implemented shape:
- common planning interface:
  - get_next_action(state, legal_actions) -> action index
  - get_turn_plan(state, legal_actions) -> list of action indices, mostly for compatibility
- keep FakePlanningAgent for smoke tests and deterministic low-quality behavior
- use HeuristicPlanningAgent for normal local AI play
- later add optional policy variants:
  - fake
  - heuristic
  - search
  - llm-assisted
  - learned

Important execution behavior:
- use one-action-at-a-time planning for the heuristic agent
- after each action is applied, regenerate legal actions and score again
- this avoids stale action-index plans after board, DON, hand, or battle state changes
- the web cockpit AI loop asks the planning agent for the next action repeatedly until it chooses end_turn, hits a prompt, wins/loses, or reaches a safety action cap

Scoring output:
- every candidate action should receive:
  - score
  - reasons
  - risk flags if useful
- the chosen action and top alternatives should be recorded in AI debug history
- the cockpit should show the score breakdown so the user can understand why the AI acted

First-pass scoring categories:
- play_card:
  - reward developing board
  - reward efficient DON usage
  - reward strong power/cost bodies
  - reward blockers or defensive pieces when behind
  - penalize low-impact plays that consume DON needed for stronger attacks
- attach_don:
  - reward reaching useful attack thresholds
  - reward pushing leader damage
  - penalize attaching to summoning-sick cards unless the attachment has another purpose
  - penalize overcommitting all DON to one low-value body
- attack:
  - reward legal leader attacks, especially with the leader because board risk is low
  - reward attacks that KO rested characters cleanly
  - reward pressure at 6000/7000/9000+ thresholds
  - penalize attacks into obvious unfavorable counter ranges
  - penalize attacking with important blockers or value pieces when the payoff is weak
- end_turn:
  - heavily penalize while good plays or attacks remain
  - choose when no positive-score action remains

First milestone status:
- HeuristicPlanningAgent can:
  - play reasonable legal cards from hand
  - attach DON to sensible attackers and threshold breakpoints
  - use leader attacks when legal and valuable
  - attack profitable rested characters
  - choose between board pressure and life pressure
  - preserve blocker-like/value attackers when behind unless the payoff is strong
  - end turn when no useful actions remain
  - expose score, reason, risk flag, summary, lookahead, and rollout debug output in the cockpit/operator paths
- the web cockpit supports heuristic mode and replans after each action
- current AI-deck effect inventory is tracked, and the refreshed 51-card AI deck currently has no manual-required entries in the inventory report
- the active AI leader is OP12-081 Koala; the player/default leader is OP09-062 Nico Robin
- generic Banish damage support is implemented for cards that actually have Banish, so successful Nico Robin life damage trashes life and skips triggers while Koala leader attacks do not Banish
- scripts/effect_inventory.py provides a repeatable inventory pass over the active deck so supported, static, partial, and manual-required effects are visible before implementing more card text
- latest targeted additions include OP07-085 Stussy, EB04-058 Borsalino, and EB03-056 Belo Betty as structured engine effects
- heuristic-owned effect choices are implemented for search/add-to-hand, discard costs, KO targets, and play-from-hand/trash style prompts
- effect choices are recorded in effect_choice_history with options, chosen card, scores, and reasons so the data can later support supervised learning
- opponent-response awareness now includes visible counter risk, active blocker pressure, known/face-up trigger risk, and crack-back risk after resting valuable characters
- shallow rollout scoring is implemented:
  - approximate action simulation
  - state evaluation over life, board, hand, blockers, and ready attackers
  - best-follow-up bonus
  - rollout_score debug output for training/evaluation traces
- default AI blocking is value-based:
  - survive blocks when possible
  - avoid early low-payoff chump blocks
  - sacrifice low-value blockers when life is low
  - sacrifice blockers to protect valuable rested characters when the trade is worthwhile

Safety rails:
- never choose actions outside get_legal_actions
- never resolve unsupported card text by heuristic or LLM guess
- never continue planning through a pending web prompt
- cap actions per AI turn
- log every decision and score breakdown
- keep deterministic behavior by default so bugs are reproducible

Remaining heuristic work:
- tune weights against longer real games instead of only focused test scenarios
- expand effect-choice scoring as more effects expose explicit legal choices
- replace approximate opponent-response heuristics with better response estimates as hidden-information and matchup data improve
- deepen rollout/search carefully, especially around multi-action sequencing, opponent turn simulation, and effect-driven branches
- add matchup/deck-specific scoring knobs once more decks are supported
- improve cockpit diagnostics for why the agent chose one line over close alternatives
- export clean decision logs that can become training data for future learned policies
- add batch evaluation: fixed scenario suite, AI-vs-AI smoke games, and win-rate/blunder metrics against fake and older heuristic baselines


Future Learned Agent Direction
Status: Feasible later / Not the next core dependency

Core idea:
- use human games, heuristic AI games, and AI vs AI self-play logs to train an agent that improves over time
- the learned agent should still receive engine-generated legal actions and rank those actions; it should not generate arbitrary free-form moves
- the engine remains the source of truth for legality, effects, state mutation, and prompts
- the heuristic agent becomes the baseline, teacher, and regression opponent for early learning work

Recommended learning path:
- first collect high-quality decision logs:
  - public game state features
  - hidden-zone-safe private features for the acting player
  - legal action list
  - chosen action
  - heuristic scores/reasons when available
  - actor source, such as human, heuristic, fake, or future learned
  - outcome labels such as damage dealt, board swing, turn result, game result, and final winner
- start with an action-ranking policy/value model over structured features
  - gradient boosted trees or a small neural MLP are better first models than a transformer
  - model output should score legal action candidates, not produce text
- later add a policy/value network that can support shallow search or MCTS-style planning
- consider transformer-style models only after the project has a large dataset and a clearer need to encode card text, long histories, or broad deck diversity
- evaluate learned agents with fixed scenario tests, holdout logs, win rate against fake/heuristic baselines, and blunder checks before letting them drive normal play

Why not an LLM or transformer first:
- the game state and legal actions are structured, so a structured action-ranker is simpler, cheaper, and easier to debug
- transformers need much more data and are most useful when sequence/text understanding is the bottleneck
- early project value comes from better logging, legal action quality, effect coverage, and evaluation harnesses, not from model size

6. Technical Risks

Main risks:
- splitting core logic across glat_engine.py and referee.py can become confusing
- letting the LLM do too much rules reasoning will create illegal-state risk
- human free-text input will be messy unless constrained first
- adding card effects without a clear effect framework will create technical debt
- reactive battle timing can become brittle if layered onto the current simplified attack flow without a clearer step model
- physical_reported mode can create state drift if reported facts are accepted without good confirmation, correction, and audit trails
- allowing any card ID from the full database can mislead users unless unsupported effects are clearly marked and manually resolved
- building a web UI can accidentally turn into a broad rewrite unless the first pass is deliberately thin and reuses the existing Python engine/session/command code
- a polished board UI is less important than a trustworthy command/chat loop; avoid over-investing in drag-and-drop or visual polish before physical play is smooth

Guiding principle:
- correctness > intelligence
- engine authority > model creativity
- structured input > free-form parsing
- physical table authority for human reports, engine authority for AI actions

That is the point where GLAT becomes a usable real-world training tool instead of only a simulation prototype.

- GLAT is now close to that threshold for the currently supported deck/testing scope
- the engine is no longer the obvious primary bottleneck

Longer-term success looks like:

"A human can play any real physical deck against the AI with low-friction reporting through the web cockpit, while the same engine can also run digital test games and future AI vs AI games for testing and training."
